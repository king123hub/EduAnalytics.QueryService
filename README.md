\# EduAnalytics.QueryService - 教育数据智能分析查询服务



\## 项目概述

本POC抽取自某区域教育质量监测系统的核心智能查询服务，展示如何通过NL2SQL实现多维度教育数据的交叉分析。



\## 业务背景

原系统服务于某市教育局，需对全市百万学生、亿级作答记录进行实时分析。核心痛点：业务方（教研员/校长/老师）不懂SQL，却需要自由组合空间、时间、内容三个维度进行交叉分析。例如：

\- "朝阳区2024年上学期八年级数学的平均分是多少"

\- "为什么朝阳区的数学平均分低于全区平均分"



\## 技术架构

用户问题 → \[大模型解析] → 结构化JSON → \[SQL生成器] → SQL → \[宽表查询] → 结果

&#x20;               ↑                           ↑                     ↑

&#x20;           意图识别                    维度映射              性能优化

&#x20;         (分析类型/维度/指标)       (字段名→业务含义)      (预聚合宽表)



\## 核心功能

\- ✅ 多维度建模：空间、时间、内容、统计四大维度

\- ✅ NL2SQL解析：模拟大模型将自然语言转为结构化意图

\- ✅ SQL生成器：根据意图生成优化的查询SQL

\- ✅ 智能下钻：问"为什么"时自动在内容维度下钻

\- ✅ 一键运行：Docker Compose快速启动



\## 架构决策与权衡



| 决策 | 方案 | 权衡 |

|------|------|------|

| 查询引擎 | 宽表 + NL2SQL vs 向量数据库 | 牺牲语义搜索灵活性，换取精确聚合确定性 |

| NL2SQL方式 | 大模型解析为JSON vs 直接生成SQL | 增加中间层开销，获得安全性和可控性 |

| 数据模型 | 宽表 vs 星型模型 | 牺牲维度扩展性，换取查询性能 |

| 下钻实现 | 预定义下钻路径 vs 动态解析 | 牺牲路径灵活性，保证分析逻辑可解释性 |



\## AI协作说明

本项目使用Cursor辅助生成了部分基础设施代码（如DbContext配置、Controller骨架）。关键的业务逻辑（维度映射规则、下钻路径算法、SQL生成策略）均由人工主导设计和实现。



AI生成代码后，人工重点调整了：

1\. 维度映射的配置化设计（从硬编码改为可扩展配置）

2\. SQL生成器的参数化查询（防止SQL注入）

3\. 意图解析器的JSON Schema结构（确保覆盖所有分析场景）
